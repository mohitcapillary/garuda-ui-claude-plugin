/clarify-prd

/clarify-prd

The first and most critical step. The AI reads the raw PRD and the linked Figma design simultaneously, then surfaces every ambiguity, mismatch, and missing decision as a structured question log — before a single line of code is planned. This prevents rework caused by spec gaps discovered mid-build.

Design Figma vs PRD mismatches

Compares every requirement in the PRD against what is actually shown in Figma. Flags when the design shows a completely different layout or feature than the spec describes.


Example found: PRD described a filterable card listing; Figma showed a columnar comparison table with 5 tabs — fundamentally different.

Spec Gap Underspecified requirements

Flags vague or missing information that would block engineers from building — e.g., no scroll behavior defined, no status filter placement shown, no edit icon interaction specified.


Example found: PRD listed status filters but gave no placement — filter bar? Dropdown? Pills? Designer had to decide.

Backend Missing API contracts & field mismatches

Checks whether API endpoints for all data requirements are documented, then cross-references payload fields against Figma screens in both directions: fields shown in Figma but missing from BE payload, and fields in BE payload with no Figma rendering. Also flags any table, list, or grid whose endpoint lacks pagination (offset/limit, page/pageSize, or cursor).


Example found: No tier listing or benefit endpoints documented. Team decided to use dummy data until BE provides contract.

📋 Real output — tier-management-clarifications.md
14 questions generated
Q1 Design KPI summary bar is absent from Figma — should it be built?
PRD specifies 4 KPI cards (total tiers, active tiers, etc.) but the Figma shows none. The only member count visible is per-tier inside the table row. PM answered: Don't show the bar, go with Figma design.
Q2 Design PRD describes a "tier listing" with status filters, but Figma shows a side-by-side comparison table — which is the primary view?
These are fundamentally different layouts. PRD implied a traditional filterable list; Figma shows a columnar comparison matrix. PM answered: Follow Figma — build the columnar comparison view with horizontal tabs.
Q8 Spec Gap Where do the status filters (Draft, Pending approval, Active, Deactivated) appear in the UI?
PRD listed filter-by-status as a feature, but neither PRD nor Figma showed placement. Designer answered: Let's not build any such filter — it's not in the Figma listing page.
Q11 Backend Which API endpoints serve tier listing, tier details, and tier benefits data?
Existing codebase has a /loyalty/api/v1 domain and a tiers.js utility but no specific tier endpoints. BE answered: Create dummy data as per Figma design; will replace once BE provides details.
Q14 Architecture Should the tier listing page live inside the existing garuda-ui app or be a standalone micro-frontend?
Existing app covers only promotion pages. Tiers is a new domain. Tech Lead answered: Create a separate route inside garuda-ui (e.g., /tiers/list).
💡
Why this step exists
Discovering that the PRD and Figma describe completely different UIs before the HLD is written saves days of rework. In this case, the AI caught 14 decision points — any one of which, if assumed incorrectly, would have caused a complete rebuild.

🏗️ Speckit format sections generated

  • Overview — 1-paragraph description of the feature and layout
  • Problem Statement — concrete problem this solves
  • Goals (G1–G5) — measurable objectives
  • Out of Scope (OS-1–OS-6) — explicitly excluded work with resolution notes
  • User Scenarios (US-1–US-6) — with acceptance criteria
  • Functional Requirements (FR-1–FR-21) — traceable to Q# decisions
  • Key Entities — data model definitions
  • Success Criteria (SC-1–SC-5) — measurable outcomes

🔗 Traceability annotations

Every requirement carries a resolution source tag:

[RESOLVED by Designer -- Q8] — Designer decision absorbed

[INFERRED] — Safe default applied by AI

/generate-hld

/generate-hld

Generates a 16-section High-Level Design document — the complete technical blueprint that maps every business requirement to React components, Redux state, API contracts, architecture alignment, risk analysis, and Figma naming. Simultaneously runs figma-node-mapper to classify every Figma node as EXACT / PARTIAL / BESPOKE. The HLD is the single source of truth for the code agent.

🎯 Why rename Figma frames?

Figma designers often name frames descriptively for design purposes (e.g., "Frame 312", "Tier column - active state"), but these don't map cleanly to React component names. The rename step bridges this gap so the code agent can look up a frame by its component name.
